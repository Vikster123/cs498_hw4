Q1: FoundationDB is built on a two-level architecture which makes the transaction layer and storage layers independent from each other‚ and hence scalable and fault tolerant separately․ The Transaction Management System (TMS) which does conflict resolution‚ version selection and transaction commit and the Storage System which stores key-value pairs and handles reads and writes run as separate processes․ Separating this allows FDB to implement the transaction logic in a modular way without coupling it to data placement․

Q2: The Resolver is the component of the database that detects read-write conflicts between transactions that are running in parallel․ The Resolver is involved on the commit path after the transaction has been tagged with a proposed commit version‚ but before it is actually committed․ The resolver checks the read conflict ranges of the transaction against the writes of transactions that committed later to determine whether the transaction must abort․
Q3: The read version tells a transaction which consistent snapshot of the database it should read from․ Thereafter‚ all reads in that transaction are logically evaluated against that same version‚ so that a transaction sees a stable view of the database even while other transactions are committing․ This is why FoundationDB provides strict serializable behavior: it can validate that the assumptions the transaction made based on that snapshot are still true at the time of commit․

Q4: Snapshot reads do not add conflict ranges to an object to prevent transaction termination due to writing data that was read by that snapshot․ Regular reads add conflict ranges‚ such that the reads do conflict‚ and this also ensures that if a transaction depends on these values‚ it is correct․ Snapshot reads are a better choice where conflicts are expensive (observation or scanning)‚ while reads without snapshot are a better choice when transaction logic must be isolated from other changes․

Q5: Recovery is cheap because FoundationDB separates durability from reads: mutations are durably committed on LogServers but are pulled into the StorageServers' states in the background․ The redo log being stored on LogServers enables storage servers to skip the giant redo log during recoveries‚ and so make transaction-state processes recover before catching up․ In the critical path of a StorageServer recovery‚ storage servers can be brought fully online before catching up with their logs․
